# Improve pi-eforge footer/status so active builds and plan counts are accurate

## Problem / Motivation

The Pi eforge footer/status currently misrepresents active build state when more than one build is running.

Symptoms observed by user screenshot/report:

- The monitor UI shows two active/running builds.
- The Pi footer shows status for only one build because `getPreferredRun()` selects a single newest/preferred session.
- The footer shows `0/1 plans` even though both active builds have two generated plans each. Code inspection confirms `/api/run-summary/:id` seeds `summary.plans` from `plan:build:start` events, so generated-but-not-started plans are absent from the denominator.

Why it matters:

- Users cannot trust Pi's always-visible status line for concurrent eforge activity.
- The current denominator communicates the wrong concept: started plans, not total planned work.
- `eforge_status` shares the same single-preferred-session behavior, so explicit status checks can also hide concurrent builds.

Evidence:

- `packages/pi-eforge/extensions/eforge/index.ts`: `formatBuildFooter()` counts completed plans over `summary.plans.length`; `getPreferredRun()` selects one run.
- `packages/monitor/src/server.ts`: `/api/run-summary/:id` constructs `plans` only from `plan:build:start`, then overlays completion/failure.
- `planning:complete` already exists and carries the generated plan list used elsewhere for session metadata.

Context evidence gathered:

- `packages/pi-eforge/extensions/eforge/index.ts` currently formats the footer via `formatBuildFooter(summary)`. The `N/M plans` footer counts `completed` statuses over `summary.plans.length`.
- The Pi extension currently selects a single status target through `getPreferredRun(cwd)`, which returns the newest running run with a `sessionId`, otherwise the newest run with a `sessionId`. The same helper is used by footer polling, `eforge_status`, and active-build stop checks.
- `packages/monitor/src/server.ts` implements `/api/run-summary/:id`. Its `plans` array is seeded only from `plan:build:start` events, then overlaid by `plan:build:complete` and `plan:build:failed`. Therefore the denominator means "plans that have started building", not "plans generated by planning".
- `planning:complete` already carries the authoritative generated plan list. Existing daemon metadata code in `packages/monitor/src/db.ts#getSessionMetadataBatch()` already derives `planCount` from `planning:complete`, and monitor code elsewhere reads compiled plans from the same event.
- `packages/client/src/types.ts#RunSummary` currently restricts plan status to `running | completed | failed`; representing generated-but-not-started plans requires adding `pending` or equivalent.
- Roadmap alignment: `docs/roadmap.md` emphasizes daemon authority, observable orchestration, TypeBox wire-schema maturity, and integration maturity. This change fits integration/observability polish and should keep daemon/client wire shapes authoritative.

Early assumptions / unknowns:

- Assumption: generated-but-not-started plans should be shown as `pending` in run summaries. Confidence high; evidence is the current UX bug and existing queue/status vocabulary, but this should be confirmed as the desired label.
- Assumption: multi-build Pi footer should aggregate running sessions rather than showing one preferred session. Confidence high; directly supported by the screenshot/user report.
- Unknown: whether `eforge_status` should return a backward-compatible single-summary object plus a `builds` array, or switch to a multi-summary response when multiple sessions are running. This has API/UX compatibility impact and needs a design decision.
- Unknown: whether `checkActiveBuilds()` should mention multiple active builds. It currently only checks the preferred run, so it may incorrectly allow daemon stop if the preferred run is not running but another is.

## Goal

Make the Pi eforge footer and `eforge_status` accurately and visibly reflect all concurrent active builds, with run-summary plan counts that match the authoritative set of generated plans rather than only those that have begun building.

## Approach

### Design Decisions

1. **Run-summary plan semantics**
   - Decision: `RunSummary.plans` should mean all generated plans for the session when known, not only plans that have started building.
   - Representation: generated-but-not-started plans use `status: 'pending'`.
   - Rationale: This makes the denominator in `N/M plans` match the user's mental model and existing monitor "2 plans" labels.
   - Overlay order: seed from latest `planning:complete`; apply `plan:build:start` as `running`; apply `plan:build:complete` as `completed`; apply `plan:build:failed` as `failed`.
   - Edge case: if no `planning:complete` exists (older sessions, malformed events, still planning), continue deriving from build events so existing sessions do not lose visibility.

2. **Multi-build detection**
   - Decision: detect active builds by grouping `/api/runs` entries with `status === 'running'` and a `sessionId`, deduplicated by sessionId.
   - Rationale: runs are the daemon's current source for active sessions; summary endpoint remains per-session.
   - Assumption: concurrent builds have distinct session IDs; validated by code naming and recorder tests for concurrent correlation, but not yet tested against a live daemon in this plan.

3. **Footer formatting**
   - Decision: preserve current single-build footer style when exactly one build is running.
   - Decision: use aggregate style when more than one build is running, e.g. `eforge builds: 2 running - 0/4 plans - 2 active - 13m`.
   - Numerator: completed plans across all running summaries.
   - Denominator: all known generated plans across all running summaries.
   - `active`: count of plans currently `running` across all summaries.
   - Duration: use oldest active build duration, or compactly show elapsed time since the oldest start. This should be documented in code to avoid ambiguity.
   - Rationale: footer space is limited; aggregate information is more useful than a lossy selected build.

4. **`eforge_status` response shape**
   - Preferred decision: return the existing single-summary shape when exactly one relevant session exists, and return a multi-session shape when multiple running sessions exist, such as `{ status: 'running', builds: RunSummary[], ...versions }`.
   - Rationale: minimizes impact for the common case while surfacing concurrency.
   - Tension: dynamic response shapes make consumers branch. Alternative is always returning `builds` and optionally retaining top-level summary fields for backward compatibility.
   - Needs confirmation: whether preserving a stable top-level single-summary shape is important for current MCP/Pi callers.

5. **Queue footer interaction**
   - Decision: `formatQueueFooter(queueItems, hasRunningBuild)` should treat `hasRunningBuild` as true if any running build exists. It can continue filtering running queue items from the queue footer to avoid duplicate running indicators.

6. **Stop safety**
   - Decision: `checkActiveBuilds()` should use any running build, not preferred build. If multiple are active, message should say e.g. `2 eforge builds are currently active. Use force: true to stop anyway.`

### Code Impact

Likely files/modules to change:

- `packages/monitor/src/server.ts`
  - `/api/run-summary/:id` should seed its `plans` map from the latest `planning:complete` event before overlaying build lifecycle events.
  - Existing nearby code already fetches session events and computes current phase/agent/event counts/duration.

- `packages/client/src/types.ts`
  - Extend `RunSummary.plans[].status` to include `pending` for generated-but-not-started plans.

- `packages/client/src/api-version.ts`
  - Bump `DAEMON_API_VERSION` if adding `pending` to the daemon HTTP wire shape is considered a breaking or meaningful additive contract change. This is probably warranted because clients that exhaustively switch on plan status should notice the new value.

- `packages/pi-eforge/extensions/eforge/index.ts`
  - Replace or complement `getPreferredRun()` with a helper that returns running sessions/runs, deduplicated by `sessionId`.
  - Update footer refresh to fetch summaries for all active running sessions.
  - Add formatting for one active build vs multiple active builds.
  - Update `eforge_status` execution/rendering to expose/render all running summaries rather than hiding all but one.
  - Update `checkActiveBuilds()` to detect any running summary.

- `packages/eforge/src/cli/mcp-proxy.ts`
  - Keep Claude/MCP-facing `eforge_status` in sync with Pi behavior where technically feasible. It currently uses `apiGetLatestRunFromRuns()` and returns one summary, so it has the same hiding behavior for concurrent sessions.

- `packages/client/src/api/queue.ts`
  - Consider adding shared helpers such as `apiGetRunningRuns()` / `apiGetRunningSessionSummaries()` to avoid duplicating run filtering in Pi and MCP. Current helpers include `apiGetRuns()`, `apiGetRunSummary()`, and `apiGetLatestRunFromRuns()`.

- Tests:
  - Add/extend monitor server tests for `/api/run-summary/:id` to verify `planning:complete` plans appear as `pending` before build start and status overlays work.
  - Add client type/wire tests if there are compile-time or schema assertions around `RunSummary`.
  - Add unit coverage for shared status aggregation/formatting if formatter logic is extracted from Pi/MCP into a testable shared helper. There are no existing `packages/pi-eforge` tests, so extracting pure helpers may be cleaner than testing extension registration directly.

### Existing Patterns / Constraints

- AGENTS.md says daemon HTTP client, route contracts, and response types are owned by `@eforge-build/client`; avoid inline path literals and duplicate wire shape definitions.
- AGENTS.md says Pi and Claude Code integration packages should stay in sync for consumer-facing behavior.
- Daemon DB read paths should use named row-to-wire helpers for run shapes; `/api/run-summary/:id` currently shapes a summary object locally, so changes should stay aligned with `RunSummary` in `@eforge-build/client`.

## Scope

### In Scope

1. Correct plan denominator semantics for run summaries:
   - `/api/run-summary/:id` should include all generated plans from the latest `planning:complete` event.
   - Generated-but-not-started plans should be represented as a non-terminal status, likely `pending`.
   - Build start/complete/failed events should overlay those generated plans.

2. Make Pi footer polling multi-build aware:
   - Detect all running sessions for the current cwd, not only one preferred run.
   - When exactly one build is running, preserve the current detailed footer style where practical.
   - When multiple builds are running, show an aggregate summary that makes concurrency obvious, e.g. `eforge builds: 2 running - 0/4 plans - 2 active - 13m`.
   - Keep queue footer behavior sensible so running queue items are not double-counted as pending queue work.

3. Make `eforge_status` multi-build aware:
   - Avoid hiding concurrent active builds.
   - Render all running builds in a compact default view and provide useful expanded detail.

4. Update shared client wire types/tests:
   - Add the pending plan status to `RunSummary` types.
   - Add or update tests around run-summary plan seeding and Pi formatting helpers if available.

5. Revisit active-build daemon-stop safety:
   - `checkActiveBuilds()` should detect any running build, not only the preferred run.

### Out of Scope

- Changing monitor UI behavior unless required by the shared wire type.
- Changing queue scheduling semantics or allowing build cancellation/reordering.
- Reworking event schemas beyond the minimal wire type change needed for run summary status.
- Adding a new daemon endpoint unless existing `/api/runs` + `/api/run-summary/:id` is insufficient.

## Acceptance Criteria

1. **Run-summary plan counts**
   - Given a session with a `planning:complete` event containing two plans and no `plan:build:start` events, `/api/run-summary/:id` returns both plans with `status: 'pending'`.
   - Given one of those plans has emitted `plan:build:start`, the summary returns two plans: one `running`, one `pending`.
   - Given lifecycle completion/failure events, the matching plan statuses become `completed` or `failed` without dropping other generated plans.
   - If no `planning:complete` event is available, run-summary continues to derive visible plans from build events for backward compatibility.

2. **Pi footer**
   - With exactly one running build, the footer remains detailed and shows completed generated plans over total generated plans, e.g. `0/2 plans` before any plan completes.
   - With multiple running builds, the footer makes concurrency explicit, e.g. includes `2 running` and an aggregate plan count such as `0/4 plans`.
   - The aggregate footer does not silently select one build when more than one is running.
   - Queue footer still shows pending/waiting non-running queue work and does not misleadingly duplicate running builds.

3. **`eforge_status`**
   - Explicit status checks reveal all active running builds rather than only one preferred/latest build.
   - The rendered Pi status view shows each active build at least by session/title/command, status, plan progress, current activity if available, and errors if any.
   - The MCP/Claude-facing `eforge_status` is updated consistently or a documented reason is provided if parity is not feasible.

4. **Stop safety**
   - Daemon stop without force is blocked if any build is running, including when multiple sessions exist and the newest/preferred run is not the only active one.

5. **Validation**
   - `pnpm type-check` passes.
   - Relevant tests pass, including new/updated tests for run-summary pending plans and multi-build aggregation/status behavior.
   - If `DAEMON_API_VERSION` is bumped, the version comment documents the `RunSummary.plans[].status = 'pending'` addition.
