# Replace presets with session-aware Copy from <tier> options

## Architecture Context

The `/eforge:profile:new` command is implemented in two consumer packages that must stay in sync per the AGENTS.md contract:

- **`packages/pi-eforge/extensions/eforge/profile-commands.ts`** — native Pi extension that drives an interactive overlay-based wizard. This is where the runtime tier-picker logic lives.
- **`eforge-plugin/skills/profile-new/profile-new.md`** — Claude Code plugin skill markdown that documents the same flow for the Claude Code path.

The wizard walks tiers in `TIER_ORDER = ['planning', 'implementation', 'review', 'evaluation']` and currently offers, for every tier:

1. Three claude-sdk-only presets (`max` / `balanced` / `fast`).
2. A single `Copy from <prevTier>` option that only references `TIER_ORDER[i - 1]`.
3. A `Custom` fallback.

This is broken for multi-harness profiles: presets imply claude-sdk is best, and on the `review` tier only `Copy from implementation` shows even when `planning` was already configured this session.

## Implementation

### Overview

Delete the preset shortcut system entirely and replace the single previous-tier copy entry with one entry per already-configured tier, generated by iterating `tierSelections` in `TIER_ORDER` order. The first tier sees only `Custom`. Each subsequent tier sees `Copy from <tierName> (<modelId>)` for every prior tier configured this session, plus `Custom`. Mirror the behavior change in the Claude Code plugin skill markdown and bump the plugin version.

### Key Decisions

1. **Remove presets entirely, do not replace them.** The PRD is explicit: presets are misleading for multi-harness use and there is no useful generic starting point. Cutting them avoids resurrecting the same problem with new labels.
2. **Session-only scope.** No cross-profile lookup. The list of copy-source tiers is derived purely from `tierSelections` accumulated this session.
3. **Preserve the Custom-fallback seed UX.** The Pi flow currently seeds `defaultHarness` and `defaultProvider` from the immediately previous tier so users do not have to re-pick provider when staying on the same harness. Keep that seeding using `TIER_ORDER[i - 1]` as the source rather than the broader `tierSelections` map — only the choice list changes shape, not the per-field defaults inside the Custom sub-flow.
4. **Choice value naming.** Use `__copy_<tierName>__` (e.g. `__copy_planning__`, `__copy_implementation__`) so the handler branch can look up `tierSelections[tierName]` directly without parsing.
5. **Plugin version bump.** Per AGENTS.md, any change inside `eforge-plugin/` requires a plugin.json version bump. The npm package version in `packages/pi-eforge/package.json` is NOT touched (release flow owns it).
6. **No CHANGELOG edit.** Per the global feedback note `feedback_changelog_managed_by_release.md`, the release flow owns CHANGELOG.md. This plan does not modify it.

## Scope

### In Scope

- Replace the `PresetName` type, `PRESETS` constant, and the preset/copy-prev choice construction + handler branches in `handleProfileNewCommand`.
- Update the Claude Code plugin skill markdown to describe the new flow (preset removal + per-prior-tier copy entries).
- Bump `eforge-plugin/.claude-plugin/plugin.json` patch version (`0.20.1` → `0.20.2`).

### Out of Scope

- Cross-profile lookup of any kind.
- Reintroducing presets under a different name or shape.
- Editing `packages/pi-eforge/package.json` version (release flow owns it).
- CHANGELOG.md edits.
- Any change to `pickCustomTier`, `loadModelsList`, `buildYamlPreview`, the daemon payload builder, or the Step 0 / Step 1 / Step 3 / Step 4 / Step 5 sections of the wizard. Only Step 2 (tier configuration) changes.
- Automated tests for the interactive overlay flow — Pi overlay UX is exercised manually per acceptance criteria; the existing test suite has no coverage for this path and adding harness-style tests is out of scope for this fix.

## Files

### Modify

- `packages/pi-eforge/extensions/eforge/profile-commands.ts` — Delete `PresetName` type and `PRESETS` constant (current lines 159-167). In `handleProfileNewCommand` tier loop (current lines 354-424):
  - Remove the `prevTier` and `prevSelection` local variables where they are used to build the choice list.
  - Drop the three `__preset_*` choice items.
  - Replace the single `__copy_prev__` entry with a loop over `TIER_ORDER.slice(0, i)` that emits one `__copy_<tierName>__` entry per tier present in `tierSelections`. Each entry uses the existing `prevDesc`-style description format (`harness · provider · modelId · effort: <effort>`, with `provider` filtered out when undefined). The label is `Copy from <tierName> (<modelId>)`.
  - Remove the `__preset_max__` / `__preset_balanced__` / `__preset_fast__` / `__copy_prev__` branches in the choice handler. Add a single `choice.startsWith('__copy_')` branch that extracts the tier name from the choice value and spreads `tierSelections[tierName]` into a fresh `selection` object.
  - For the `Custom` fallback's `defaultHarness` / `defaultProvider` seed, derive a local `prevTier`/`prevSelection` pair from `TIER_ORDER[i - 1]` (only used inside the Custom branch). Keep the existing fallback chain: `prevSelection?.harness ?? (name.startsWith('claude-') ? 'claude-sdk' : 'pi')` and `prevSelection?.provider`.
- `eforge-plugin/skills/profile-new/profile-new.md` — In Step 2 (current lines 31-50):
  - Delete the entire **Preset shortcuts** subsection (the `max` / `balanced` / `fast` bullets and the heading line).
  - Rename **Copy from previous tier** to **Copy from a previously configured tier** and update its body: "From the second tier onward, present one `Copy from <tierName> (<modelId>)` entry per tier already configured in this session (e.g. on `review`: `Copy from planning` and `Copy from implementation`, in `TIER_ORDER` order). The default selection is copy-from-immediately-previous when the user just presses enter."
  - Replace the line that reads `For the **planning** tier there is no "copy from previous" option — present presets or custom only.` with: `For the **planning** tier present only **Custom**, since no prior tier exists yet.`
  - Leave the **Custom** sub-flow bullets, Step 3 / 4 / 5 sections, and the YAML examples unchanged.
- `eforge-plugin/.claude-plugin/plugin.json` — Bump `version` from `0.20.1` to `0.20.2`.
